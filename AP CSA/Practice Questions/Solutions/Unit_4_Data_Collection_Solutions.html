Q1MEDIUMAP MClength vs size

arr.length vs list.size()?

  • (A) 4 3
  • (B) 3 3
  • (C) 4 4
  • (D) Compile-time error.
Answer: (A)
Two collection types, two different interfaces:
  • Arrays use a field: arr.length (no parentheses). The literal {1,2,3,4} has length 4.
  • ArrayList uses a method: list.size() (with parentheses). After three adds, list.size() == 3.
Trap (D): arrays do have length — but they do not have .length(). String, the other way around, uses s.length(). Mix these up and the compiler instantly complains. AP MCQ writers love this comparison.
Q2MEDIUMAP MCOut-of-Bounds

int[] vals = {10,20,30}; println(vals[3]);

  • (A) 0 (default).
  • (B) 30 (last).
  • (C) ArrayIndexOutOfBoundsException.
  • (D) Compile-time error.
Answer: (C)
An array of length n has valid indices 0 through n − 1. With vals.length == 3, valid indices are 0, 1, 2. Reading vals[3] is out of bounds and Java throws ArrayIndexOutOfBoundsException at run time.

Trap (D): the bounds check is dynamic, not static — the compiler cannot generally tell that an index will be out of range. Trap (B): Java does not "wrap around" or silently clamp to the last element — some languages do this, but Java does not.

Q3MEDIUMAP MCEnhanced for-loop

for (int v : vals) v *= 10; — does it change vals?

  • (A) 10 40
  • (B) 1 4
  • (C) 1 40
  • (D) Compile-time error.
Answer: (B)
The enhanced-for loop variable v is a copy of each array element (since int is a primitive). Reassigning v changes only the local copy — the array is untouched.
  • To modify in place, use an indexed loop: for (int i = 0; i < vals.length; i++) vals[i] *= 10;
  • Even if the element type were an object, reassigning v wouldn't modify the array — you'd need to mutate the object's state via v.someMethod(...).
Trap (D): Java allows reassignment of the for-each variable; it just has no external effect. This is a classic AP "looks-like-it-should-work" trap.
Q4HARDAP MCArray Reverse

Reverse {1,2,3,4,5} via swap loop — print a[0] a[2] a[4]?

  • (A) 1 3 5
  • (B) 5 3 1
  • (C) 5 4 1
  • (D) 5 2 1
Answer: (B)
a.length / 2 = 2, so the loop runs for i = 0, 1 — exactly enough swaps to reverse a 5-element array. 
i = 0: swap a[0] ↔ a[4]: {5, 2, 3, 4, 1}
i = 1: swap a[1] ↔ a[3]: {5, 4, 3, 2, 1}
i = 2: 2 < 2 false → exit
Note the middle element a[2] is untouched — for an odd-length array, the loop correctly leaves the center alone. Output: 5 3 1.

Trap (A) is the original array (no swaps happened). The endpoint computation a.length - 1 - i is the key to a clean in-place reverse — get it wrong and you blow up the array.

Q5HARDAP MCArrayList Remove + Shift

[20, 20, 30, 40]; remove every value divisible by 20 with a forward index loop.

  • (A) [30]
  • (B) [20, 30]
  • (C) []
  • (D) [20, 20, 30]
Answer: (B)
The bug is that after remove(i), every later element shifts left by one — but i still increments, so the loop skips the element that just moved into position i. 
state           i  get(i)  action
[20,20,30,40]   0    20    remove → [20,30,40], i = 1
[20,30,40]      1    30    keep,        i = 2
[20,30,40]      2    40    remove → [20,30],    i = 3
[20,30]         3    —     3 < 2 false → exit
The second 20 (which shifted into index 0) is never re-checked, so it survives in the list. Result: [20, 30].

Trap (A) is the intended result if the bug weren't there. Fix: either loop backwards (for i = list.size() − 1 down to 0), or do i-- after a successful remove so the same index is re-checked.

Q6HARDAP MC2D Access

g[1][2] + g[2][1] on a 3×3 grid 1..9.

  • (A) 8
  • (B) 11
  • (C) 13
  • (D) 14
Answer: (D)
Indexing is g[row][col], both zero-based. 
row\col   0   1   2
   0   |  1   2   3
   1   |  4   5   6
   2   |  7   8   9
  • g[1][2] → row 1, col 2 → 6
  • g[2][1] → row 2, col 1 → 8
  • Sum: 14
Trap (B) swaps the indices (reads g[2][1] as "col 2, row 1" = 6, doubling up). Trap (C) assumes g[r][c] == r * 3 + c + 1 arithmetic without actually checking the diagonal swap. Always draw the grid.
Q7HARDAP MC2D Traversal (Non-Square)

m = {{1,2},{3,4},{5,6}}; sum elements where r == c.

  • (A) 5
  • (B) 9
  • (C) 21
  • (D) 1
Answer: (A)
The array has 3 rows × 2 columns — it is not square. The inner loop bound c < m[r].length is 2 for every row, so c only ever takes values 0, 1.
  • r=0, c=0: r == c ✓ → add m[0][0] = 1 → s = 1
  • r=0, c=1: 0 ≠ 1 → skip
  • r=1, c=0: 1 ≠ 0 → skip
  • r=1, c=1: r == c ✓ → add m[1][1] = 4 → s = 5
  • r=2, c=0 and c=1: 2 ≠ 0, 2 ≠ 1 → skip both (no c = 2 visited)
Final s = 5. Trap (B) assumes square geometry and adds m[2][2], which doesn't exist. Trap (C) sums every element.

Lesson: a int[][] doesn't have to be rectangular — it's an array of arrays. Always bound the inner loop by m[r].length, not m.length.

Q8HARDAP MCArrayList Insert

Insert "X" at index 1, then "Y" at index 0.

  • (A) [a, X, b, c]
  • (B) [Y, X, a, b, c]
  • (C) [Y, a, X, b, c]
  • (D) [a, b, X, Y, c]
Answer: (C)
add(index, element) inserts at the given index and shifts the existing elements (at that index and after) one position to the right. 
start                     [a, b, c]
add(1, "X")               [a, X, b, c]
add(0, "Y")               [Y, a, X, b, c]
Trap (B) assumes both inserts happen against the original list, not the updated list. (D) appends at the end. The key is that the second add is computed against the new size 4 list, so inserting at 0 pushes everything (including X now at index 1) right by one.
Q9HARDAP MCLinear "Find" — No Break

{3,7,5,7,9,7}; loop without early exit, store every match's index in idx.

  • (A) -1
  • (B) 1
  • (C) 3
  • (D) 5
Answer: (D)
The loop has no early-exit (no break, no return), so it visits every index 0–5. Each match overwrites idx: 
i    arr[i]   action               idx after
0     3       skip                    -1
1     7       idx = 1                  1
2     5       skip                     1
3     7       idx = 3                  3
4     9       skip                     3
5     7       idx = 5                  5
Final idx = 5 — the index of the last occurrence. Trap (B) is what a correctly-written "find first" would produce (with a break after the first match). On the AP exam, watch for the missing break: it inverts the algorithm from "find first" to "find last."
Q10HARDAP MCSelection Sort — 2 Passes

{5, 3, 8, 1, 9, 2} — state after 2 complete passes of selection sort?

  • (A) {1, 2, 3, 5, 8, 9}
  • (B) {1, 2, 8, 5, 9, 3}
  • (C) {1, 3, 8, 5, 9, 2}
  • (D) {3, 5, 8, 1, 9, 2}
Answer: (B)
Selection sort: on pass k (0-indexed), find the minimum of a[k..end] and swap it to position k. 
start              : {5, 3, 8, 1, 9, 2}
Pass 1 (k=0):
  min of full array = 1 at idx 3
  swap a[0] ↔ a[3] : {1, 3, 8, 5, 9, 2}
Pass 2 (k=1):
  min of a[1..5]   = 2 at idx 5
  swap a[1] ↔ a[5] : {1, 2, 8, 5, 9, 3}
After 2 passes, indices 0 and 1 hold the two smallest values; everything from index 2 onward is whatever was left after the swaps. Trap (A) jumps to the fully-sorted state. (C) stops after one pass.
Q11HARDAP MCInsertion Sort — 2 Passes

{5, 2, 8, 1, 9, 3} — state after 2 complete passes of insertion sort?

  • (A) {2, 5, 8, 1, 9, 3}
  • (B) {1, 2, 5, 8, 9, 3}
  • (C) {2, 5, 8, 9, 1, 3}
  • (D) {5, 2, 8, 1, 9, 3}
Answer: (A)
Insertion sort: starting at index 1, insert a[i] into the already-sorted prefix a[0..i-1]. 
start         : {5, 2, 8, 1, 9, 3}
Pass 1 (i=1):
  insert 2 into [5]    →   prefix becomes [2, 5]
  array         : {2, 5, 8, 1, 9, 3}
Pass 2 (i=2):
  insert 8 into [2, 5] →   8 > 5, no shift
  array         : {2, 5, 8, 1, 9, 3}
After 2 passes, indices 0–2 are a sorted prefix; the tail is untouched. Trap (D) assumes nothing has been inserted yet.

Selection vs Insertion (compare with Q10): selection sort fixes one element at a time from the front of the unsorted region; insertion sort grows a sorted prefix by slotting each new element. After k passes both have the prefix correct, but the trailing region looks different.

Q12HARDAP MCBinary Search Trace

Binary-search {2,5,7,12,18,23,31,42,56} for 19 — last element examined?

  • (A) 18
  • (B) 23
  • (C) 31
  • (D) 56
Answer: (B)
Indices: 0:2, 1:5, 2:7, 3:12, 4:18, 5:23, 6:31, 7:42, 8:56. 
lo  hi  mid  arr[mid]  compare to 19    next
 0   8    4      18      18 < 19          lo = 5
 5   8    6      31      31 > 19          hi = 5
 5   5    5      23      23 > 19          hi = 4
 5   4   —       —       lo > hi, exit (not found)
Examined values in order: 18, 31, 23. The last one examined before the loop exits is 23.

Note the search runs even when the target is absent — the loop terminates when lo > hi. Trap (A) stops after the first comparison; (C) stops before the final mid.

Q13HARDAP MCRecursion — Factorial

f(n) = (n <= 1) ? 1 : n * f(n - 1); f(5)?

  • (A) 5
  • (B) 25
  • (C) 120
  • (D) StackOverflowError.
Answer: (C)
Standard factorial — the base case n <= 1 returns 1, the recursive case multiplies. 
f(5) = 5 * f(4)
     = 5 * 4 * f(3)
     = 5 * 4 * 3 * f(2)
     = 5 * 4 * 3 * 2 * f(1)
     = 5 * 4 * 3 * 2 * 1
     = 120
Trap (D): stack overflow only happens if the base case is never reached. Here the argument n - 1 strictly decreases each call and eventually hits 1, so the recursion terminates in 5 levels.

An AP exam might also ask: how many activations of f are on the stack at peak? Answer: 5 (one for each of f(5), f(4), …, f(1)).

Q14HARDAP MCRecursion — Halving

g(n) = (n < 2) ? 0 : 1 + g(n / 2); g(17)?

  • (A) 3
  • (B) 4
  • (C) 5
  • (D) 17
Answer: (B)
The recursion counts the number of integer halvings needed to drop below 2. 
g(17) = 1 + g(8)
      = 1 + 1 + g(4)
      = 1 + 1 + 1 + g(2)
      = 1 + 1 + 1 + 1 + g(1)
      = 1 + 1 + 1 + 1 + 0
      = 4
Conceptually, this is ⌊log₂(n)⌋ for n ≥ 1 — exactly the running-time intuition behind binary search (Q12).

Trap (C) counts the base call. Trap (A) stops one halving early. Be careful with the integer-division step: 17 / 2 = 8, not 8.5.

Q15HARDAP MCArray Aliasing in Method

swapEnds(vals) on {1,2,3,4,5} — caller sees what?

  • (A) 1 5
  • (B) 5 1
  • (C) 1 1
  • (D) No effect.
Answer: (B)
Arrays in Java are objects. The parameter a inside the method is a copy of the reference, but it points to the same underlying array as vals in the caller. Mutating the elements is visible everywhere that reference is held. 
Before: vals = {1, 2, 3, 4, 5}
Inside swapEnds:
  tmp = a[0]            // tmp = 1
  a[0] = a[4]           // {5, 2, 3, 4, 5}
  a[4] = tmp            // {5, 2, 3, 4, 1}
After:  vals = {5, 2, 3, 4, 1}
println vals[0] + " " + vals[4]: "5 1"
Trap (D): students sometimes assume "pass by value" means "no side effects." For object types, the reference is the value, and side effects on the referent are absolutely visible. Same rule applies to ArrayList and any custom class instance. Compare with Unit 3 Q14: primitives don't escape, references do.
