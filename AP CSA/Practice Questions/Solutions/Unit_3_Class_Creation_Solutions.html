Q1MEDIUMAP MCthis Keyword

What does this.size = size; do in the constructor?

  • (A) Assigns the parameter size to the instance variable size.
  • (B) Assigns the parameter to itself.
  • (C) Compile-time error.
  • (D) Creates a discarded local.
Answer: (A)
When a parameter shadows an instance variable (same name in scope), an unqualified size refers to the parameter. The prefix this. reaches past the parameter and refers explicitly to the instance variable of the current object.
  • Left of =: this.size → instance variable.
  • Right of =: bare size → parameter.
Compare with Q6: writing size = size; (no this.) self-assigns the parameter and leaves the field at its default value.
Q2MEDIUMAP MCstatic Variables

Which statement about a static variable is true?

  • (A) Each instance has its own copy.
  • (B) Auto-initialized to null regardless of type.
  • (C) A single copy is shared by all instances.
  • (D) Read-only after initialization.
Answer: (C)
static binds the variable to the class, not to any individual instance. All instances share the same single storage location.
  • Access from outside: ClassName.varName.
  • Default initialization follows the type (numerics get 0, boolean gets false, references get null) — not "null regardless of type."
  • Reassignment is allowed unless declared final.
Use case: counters that span all instances (see Q5), shared constants, factory helpers.
Q3MEDIUMAP MCDefault Constructor

A class with no declared constructors:

  • (A) Cannot be instantiated.
  • (B) Receives a default no-arg constructor.
  • (C) Fails to compile.
  • (D) Is implicitly a static utility class.
Answer: (B)
The compiler synthesizes a public no-argument constructor with an empty body — i.e. public Foo() {} — so new Foo() works and instance variables fall back to their type defaults.

Important caveat: as soon as you declare any explicit constructor (e.g. public Foo(int x)), the compiler will no longer supply the no-arg default. new Foo() then becomes a compile-time error unless you write the no-arg form yourself. This is a common AP trap.

Q4HARDAP MCObject Reference Param

bump(k); bump(k); println(k.getCount());

  • (A) 0
  • (B) 1
  • (C) 2
  • (D) Compile-time error.
Answer: (C)
In Java, parameters are passed by value — but for object types, the reference is what gets copied. The parameter c inside bump points to the same Counter object as k outside.
  • First bump(k): c.inc() mutates the shared object → count = 1.
  • Second bump(k): same object → count = 2.
  • k.getCount() → 2.
Contrast with primitives (Q14): assigning to a primitive parameter does not affect the caller. The reference-vs-primitive distinction is the highest-yield Unit 3 topic on the AP exam.
Q5HARDAP MCstatic Counter

Three new Widget() calls; print Widget.total().

  • (A) 1
  • (B) 2
  • (C) 3
  • (D) Compile-time error.
Answer: (C)
built is static — one shared counter across the entire Widget class. Each call to the constructor increments it once:
  • new Widget() #1 → built = 1 (discarded reference; the object exists)
  • new Widget() #2 → built = 2
  • new Widget() #3 → built = 3 (assigned to w)
Widget.total() returns 3. Trap (B): "I only stored one reference (w), so only one was constructed" — false. All three constructor calls execute regardless of whether you keep the reference. Trap (D): a static method can access a static field — only instance fields are off-limits (see Q12).
Q6HARDAP MCMissing this. Trap

Constructor body is x = x; (no this.); new Point(7).getX()?

  • (A) 0
  • (B) 7
  • (C) -1
  • (D) Compile-time error.
Answer: (A)
Inside the constructor, the parameter x shadows the instance variable x. Without the this. qualifier, both sides of x = x; refer to the parameter — it's a self-assignment of the parameter to itself.
  • The instance variable x is never touched, so it keeps its default value 0.
  • getX() returns the instance variable → 0.
Java does allow self-assignment without error, so (D) is wrong. Fix: write this.x = x; as in Q1. This bug is silent and embarrassingly common in beginner Java.
Q7HARDAP MCConstructor Chaining

public Box() { this(10); } — new Box().getSize()?

  • (A) 0
  • (B) 10
  • (C) Compile-time error: cross-constructor call not allowed.
  • (D) StackOverflowError from recursive constructor call.
Answer: (B)
this(...) as the first statement of a constructor invokes another constructor of the same class — this is legal "constructor chaining."
  • new Box() enters the no-arg constructor.
  • It immediately delegates to Box(10) via this(10).
  • Box(10) sets this.size = 10.
  • Control returns; the object is fully constructed. getSize() → 10.
Trap (D): the call is to a different constructor, not the same one — no recursion. Trap (C): cross-constructor calls are allowed; the rule is "must be the first statement."
Q8HARDAP MCMutator + Validation

deposit(100); deposit(-50); deposit(30); getBal()?

  • (A) 80.0
  • (B) 100.0
  • (C) 130.0
  • (D) 180.0
Answer: (C)
deposit applies the change only if amt > 0:
  • deposit(100): 100 > 0 ✓ → bal = 100.0.
  • deposit(-50): -50 > 0 ✗ → ignored.
  • deposit(30): 30 > 0 ✓ → bal = 130.0.
Trap (A) assumes the negative is treated as a withdrawal: 100 − 50 + 30 = 80. (D) takes the absolute value of every input: 100 + 50 + 30 = 180. Both are valid design choices, but neither is what this code does.

This input-validation pattern is a Unit 3 staple — the AP expects you to read the guard precisely.

Q9HARDAP MCMethod Overload Resolution

square(3) vs square(3.0) on overloaded methods.

  • (A) 9 then 9.0
  • (B) 9.0 then 9.0
  • (C) 9 then 9
  • (D) Compile-time error.
Answer: (A)
Overload resolution picks the most specific matching signature at compile time, based on the static type of each argument.
  • 3 is an int literal → exact match for square(int) → returns int 9 → printed as 9.
  • 3.0 is a double literal → exact match for square(double) → returns double 9.0 → printed as 9.0.
Trap (D): two methods can share a name as long as their parameter lists differ in number or type — this is overloading, not a conflict.
Q10HARDAP MCObject Equality

p == q vs p == r where r = p and q = new Pt(1,2).

  • (A) true true
  • (B) false true
  • (C) true false
  • (D) false false
Answer: (B)
The == operator on objects compares references, not contents.
  • p and q are made by separate new Pt(...) calls — different objects in memory, regardless of identical contents. p == q → false.
  • r = p copies the reference, so r points to the same object. p == r → true.
Rule (Unit 1/3 carryover): for "do they have the same contents?" you'd override and use .equals(...); == is "do they refer to literally the same instance?"
Q11HARDAP MCLocal Variable Shadows

drain() redeclares int level = 0; locally — final getLevel()?

  • (A) 90
  • (B) 100
  • (C) -10
  • (D) 0
Answer: (B)
The line int level = 0; declares a new local variable named level inside drain(). From that point in the method, any unqualified reference to level refers to the local, not the instance variable.
  • Local level: 0 → -10. (Then disappears when drain returns.)
  • Instance level: untouched, still 100.
getLevel() reads the instance field → 100. Trap (A) assumes the mutation reaches the instance variable. To actually mutate the instance, drop the int declaration: level -= 10; alone would resolve to the instance variable (with no shadowing local).
Q12HARDAP MCstatic Accessing Instance

public static int getCount() { return count; } where count is a non-static field.

  • (A) Compiles and works.
  • (B) Compiles; returns 0.
  • (C) Compile-time error: non-static field cannot be referenced from a static context.
  • (D) Run-time NullPointerException.
Answer: (C)
A static method is associated with the class, not an instance — so there is no implicit this. Inside it, an unqualified reference to a non-static field is ambiguous: "which instance's count?" The compiler refuses.
  • Exact message: non-static variable count cannot be referenced from a static context.
  • Fix #1: make count static (one shared counter for the class — see Q5).
  • Fix #2: make getCount() non-static, so it has an implicit this to read from.
Trap (B): the field has a value (0), but the compiler won't even let the code reach run time.
Q13HARDAP MCEncapsulation Access

p.name = "Bob"; from outside Person, where name is private.

  • (A) Compiles; sets name.
  • (B) Compiles; silently ignored.
  • (C) Compile-time error: private access.
  • (D) Run-time IllegalAccessException.
Answer: (C)
private restricts access to within the same class declaration only — not even other classes in the same package can read or write it directly. Attempting p.name = "Bob"; from outside Person is rejected at compile time.
  • Same-package alone is not sufficient (that's default / package-private, which uses no modifier).
  • Same-class is sufficient — code inside Person can read/write name freely.
  • To allow external mutation, expose a mutator method like setName(String), ideally with validation.
Trap (D): Java enforces private at compile time, so the run-time exception class isn't involved.
Q14HARDAP MCPrimitive vs Object Params

After modify(x, w) assigns x = 999 and w.n = 999: println(x + " " + w.n)?

  • (A) 5 5
  • (B) 999 999
  • (C) 5 999
  • (D) 999 5
Answer: (C)
Java is uniformly pass-by-value — but for objects, the value being passed is a copy of the reference.
  • Primitive x: the parameter inside modify is a copy of the value 5. Reassigning the local copy to 999 has zero effect on the caller's x. Caller's x stays 5.
  • Object w: the parameter inside modify is a copy of the reference — it points to the same Wrap object. The dot access w.n = 999 reaches across the reference and mutates the shared object. Caller sees w.n = 999.
Distinction: reassigning the parameter vs mutating the object the parameter points to. The first never escapes; the second does.
Q15HARDAP MCtoString Implicit Call

System.out.println("Found: " + c); where c overrides toString.

  • (A) Memory address.
  • (B) Found: (25¢)
  • (C) Found: 25
  • (D) Compile-time error.
Answer: (B)
String concatenation with + implicitly invokes toString() on the object operand. println also does this when handed a non-String object directly.
  • "Found: " + c → "Found: " + c.toString() → "Found: " + "(25¢)" → "Found: (25¢)".
Trap (A): the default Object.toString() does produce something like Coin@1a2b3c — but only when no override is provided. Here Coin overrides toString, so that default is shadowed. Designing a class for printability nearly always means writing a toString; the AP exam expects you to recognize this idiom.
